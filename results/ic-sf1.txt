LDBC SNB Interactive (IC) — SF1, single-machine head-to-head
rustychickpeas (this engine) vs Kùzu, same data, same seeds (pick_seeds).

CAVEATS: timings below are SF1 under load avg ~6.7 on 16 cores (a shared box) —
read them RELATIVELY, not as absolutes; both engines run under the same load.
Kùzu runs multi-threaded by default, our queries are mostly single-threaded.
All 20 queries are value-identical (kuzu/compare.py ALL PASS).

The Kùzu Cypher was optimized first (tasks 055-064) so this is a fair fight —
the initial reference Cypher was naive (correlated subqueries, IN-list
membership, un-deduped knows*1..2). Per-query Kùzu gains:
  IC4 6347->151 ms (~42x)   IC5 4455->434 ms (~10x)   IC6 2300->223 ms (~10x)
  IC9 1672->431 ms (~4x)    IC12 925->342 ms (~3x)    IC3 745->562 ms
And chickpeas IC14 was switched to a bidirectional search (~14->~10 ms).

Net: chickpeas wins ~14, Kùzu wins 4 (IC1/IC5/IC13/IC14). chickpeas dominates the
short reads and CSR-friendly traversals (IS*, IC7/IC8/IC10) — neighbour iteration
vs query-engine overhead. Kùzu's vectorized engine wins the native shortest paths
(IC13/IC14) and the heavy multi-hop aggregation (IC5), and ties IC1. IC10's gap is
real (the 2-hop foaf expansion + per-foaf counting is inherent to the query, not
bad Cypher). Precise magnitudes still want an unloaded machine.

Reproduce:
  cargo run --release --bin ic                     # rust timings
  .venv-kuzu/bin/python kuzu/run_ic.py sf1         # kuzu timings (db-sf1-faithful)

  .venv-kuzu/bin/python kuzu/run_ic.py sf1 --emit-json /tmp/xchk
