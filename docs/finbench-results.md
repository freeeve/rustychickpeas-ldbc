# FinBench complex reads — Rust vs Kùzu (SF10)

Head-to-head wall-clock for the 12 LDBC FinBench Transaction Complex Reads
(TCR1–TCR12, spec v0.2.0-alpha §5.1), the Rust CSR engine against Kùzu Cypher,
on the **same** generated dataset.

- **Data:** `data/finbench/raw`, generated by `scripts/gen_finbench.sh`
  (`FINBENCH_MEM=24g`). **1,103,805 nodes / 8,962,710 edges.** Both loaders agree
  on these counts exactly (task 078).
- **Load time:** Rust snapshot 11.4 s; Kùzu `COPY` import 61.8 s.
- **Rust:** `cargo run --release --bin finbench` — median of 30 runs per query
  (`harness::time_query`).
- **Kùzu:** `kuzu 0.11.3`, `kuzu/finbench_queries.py` — median of 7 runs per query.
- **Hardware:** Apple Silicon (Darwin 24.6), release builds.

**Summary.** Rust is at or below Kùzu's latency on every query, and 1–4 orders of
magnitude faster on the four where Kùzu's planner struggles: CR4 (time-ordered
3-cycle, 302 ms), CR2 (reverse-reachability, 30 ms even after unrolling the
recursion — see caveats), CR9 (laundering join, 26 ms), CR12 (14 ms). The eight
others are sub-6 ms on both engines.

## Read these numbers with the caveats

This is a **timing comparison of equivalent query patterns over identical data**,
not a row-for-row correctness oracle. Three things differ between the columns:

1. **Seeds differ.** Each engine picks its own highest-degree seed independently
   (Rust by internal `u32` id; Kùzu by original `i64` id), so the two columns do
   not run against the same start vertex. Row counts therefore are not expected to
   match — they characterize the *shape* of work, not a shared answer key.
2. **Result granularity differs.** The Rust impls return the full populated result
   (e.g. CR8 → 124 downstream accounts, CR10 → 56 co-investors); several Kùzu
   queries are written as aggregates returning a single row (`count(*)`, `sum(...)`),
   so "Kùzu rows = 1" usually means one aggregate row, not one match.
3. **Two Kùzu queries return 0 rows on their seed** (CR1, CR4): the seed Kùzu
   picked has no blocked-medium upstream / no qualifying cycle. CR1 in particular
   makes the Kùzu 5.29 ms *less* work than Rust's 5.96 ms over 605 real results —
   so CR1 is the one row where the head-to-head is *not* in Rust's favour, and the
   reason is that Kùzu is doing strictly less. (The Rust driver scans its top-500
   degree pool for a seed whose CR1 is non-empty, since the pattern is sparse —
   ~9.8 k of 439 k accounts qualify.)

CR12 returns 0 on **both** engines — confirmed a genuine data property at SF10
(no `transfer` edges originate from person-owned accounts), not a seed artifact.

### Kùzu-specific workarounds in the reference

The Cypher reference (`kuzu/finbench_queries.py`) is faithful-ish but had to route
around two engine limits, both of which inflate Kùzu's numbers above a naive
translation:

- A parameter inside a recursive-rel comprehension `all(x IN rels(e) …)` crashes,
  so time-window bounds are inlined as literals.
- Kùzu's recursive `*1..3` engine explodes on hub vertices — CR2 ran **20 s**
  recursive, so reverse-reachability is unrolled into explicit hops in the driver
  (still 30 ms). The Rust CR2 does the same reverse walk natively at <0.01 ms.

## Faithfulness pass

Beyond timing, four queries were audited against the spec and corrected so the
Rust result is a faithful answer, not just fast:

- **CR1** — seed scan added so the benchmark exercises a non-empty result (the
  blocked-medium upstream pattern is sparse). Commit `660c179`.
- **CR6** — rewritten to the spec's many-to-one shape (the card's outgoing
  withdrawals + the in-window incoming transfers preceding the last withdrawal),
  replacing a draft that read the wrong edge direction. Commit `d15180b`.
- **CR10** — reshaped from pairwise Jaccard to the spec's one-to-many
  co-investor / shared-company count. Commit `d15180b`.
- **CR5** — parallel `src->dst` transfer edges are now collapsed to one step per
  neighbour (earliest in-window timestamp — the faithful test that an ascending
  trace exists), so the result is a true *set* of traces per the spec note;
  `truncationOrder` is now honoured too. Commit `542bbbe`.
- **CR8 / CR9** — audited, faithful as-written (CR8: `amt > threshold·Σ(in-transfers
  of source)` upstream rule, `ratio = inflow/loanAmount`; CR9: `ratioRepay=Σe1/Σe2`,
  `ratioDeposit=Σe1/Σe4`, `ratioTransfer=Σe3/Σe4` with the `-1` sentinels). No change.

## Reproduce

```bash
# Rust (loads raw/, runs + times all 12 CRs)
cargo run --release --bin finbench -- data/finbench/raw

# Kùzu reference (build the DB once, then time)
.venv-kuzu/bin/python kuzu/finbench_import.py  data/finbench/raw kuzu/db-finbench-sf10
.venv-kuzu/bin/python kuzu/finbench_queries.py kuzu/db-finbench-sf10
```
