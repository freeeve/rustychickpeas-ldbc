## Notes

IC14 was already on the bidirectional `weighted_shortest_path` (Outgoing forward /
Incoming backward, avoiding the Both double-traversal) from earlier work. The
profile is dominated by that core primitive itself (~80%+) — its bidirectional
Dijkstra's internal dist HashMaps + RoaringBitmap visited set + the heap. The
per-rel `interaction.get(&(sorted pair))` tuple-hashing is only ~12%, and turning
it into an rel-pos-indexed array would need a ~56 MB side table (rel.pos is the
global CSR position) for ~1 ms — not worth it, and it wouldn't close the gap to
Kùzu's vectorized engine.

**No query-side change.** The lever is in the core primitive `weighted_shortest_path`.

### Core lever — LANDED (core task 027)

The core change shipped as `perf(core): reuse weighted_shortest_path scratch
across calls`. Maintainer signed off on the **reused-HashMap** shape rather than
dense generation-stamped buffers: the dense form would be ~2 ns/access but cost
24 B × node_count (~70 MB/thread at SF1); reusing the two dist maps + two heaps
from a thread-local scratch (`clear()` per call, keeps capacity) keeps ~1 MB/thread
and removes the per-call churn. Verified value-identical (IC13 + IC14 byte-identical);
IC14 steady-state allocations **1,081,456 B → 0**. Wall-clock confirmation pending a
quiet machine (a concurrent benchmark run pinned load at 70–165 during measurement).

**Status: done (query-side optimal; core lever landed as core task 027).**
