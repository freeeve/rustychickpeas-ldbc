## Notes

IC14 was already on the bidirectional `weighted_shortest_path` (Outgoing forward /
Incoming backward, avoiding the Both double-traversal) from earlier work. The
profile is dominated by that core primitive itself (~80%+) — its bidirectional
Dijkstra's internal dist HashMaps + RoaringBitmap visited set + the heap. The
per-edge `interaction.get(&(sorted pair))` tuple-hashing is only ~12%, and turning
it into an edge-pos-indexed array would need a ~56 MB side table (rel.pos is the
global CSR position) for ~1 ms — not worth it, and it wouldn't close the gap to
Kùzu's vectorized engine.

**No query-side change.** The real lever is a **core dense-scratch optimization of
`weighted_shortest_path`** (the `bfs_distances` treatment — replace the per-node
dist HashMaps with thread-local generation-stamped dense buffers — but for the
bidirectional case). It would speed up IC14, IC13 (now on this primitive), and any
weighted-path query. **Sign-off-gated core follow-up**, not done here.

**Status: done (query-side optimal; core `weighted_shortest_path` scratch is the follow-up).**
