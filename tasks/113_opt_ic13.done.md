# 113 — Optimize IC13: unweighted shortest path

Per-query optimization loop. Methodology + commands:
[100_ic_perf_methodology.md](100_ic_perf_methodology.md).

## Loop
- [x] 1. Bench allocations: 48 allocs / 368,608 bytes (the whole-component distance map)
- [x] 2. CPU profile: **99.97% `bfs_distances`** — a full unidirectional BFS over p1's
  entire connected component just to read one distance
- [x] 3. Optimize: `weighted_shortest_path` (bidirectional Dijkstra) with unit weights
  — meet-in-the-middle, explores ~5× fewer nodes; cost == hop count
- [x] 4. Re-bench: 37 allocs / 827,536 bytes
- [x] 5. Wall-clock A/B: ~4.2 → ~0.83 ms (~5×), same window
- [x] 6. Value-identity: IC emit byte-identical

## Measurements
| metric                  | baseline | after |
|-------------------------|----------|-------|
| allocs                  | 48       | 37    |
| bytes                   | 368,608  | 827,536 (up) |
| wall-clock median (ms)  | ~4.2     | ~0.83 (~5×) |
| hot fn (CPU %)          | full bfs_distances | bidirectional Dijkstra |

## Notes

IC13 computed `bfs_distances(p1, …, None)` — distances to **all** of p1's
component (the seed is high-degree → ~the whole network) — then read p2's. Switch
to `weighted_shortest_path(p1, p2, …, |_,_| 1.0)`: a bidirectional search that
meets in the middle, exploring ~5× fewer nodes (unit weights ⇒ path cost = hop
count). **~5× faster; now beats Kùzu** (3.7 ms) instead of losing. Value-identical.

Caveat: bytes went **up** (369 KB → 828 KB) — bidirectional Dijkstra allocates two
frontiers + priority queues vs one BFS map; transient (freed after the call), and
the wall-clock win dominates. A dedicated bidirectional *unweighted BFS* core
primitive (sign-off) would deliver the speed without the Dijkstra-structure bytes.
