# 111 — Optimize IC11: job referral

Per-query optimization loop. Methodology + commands:
[100_ic_perf_methodology.md](100_ic_perf_methodology.md).

## Loop
- [x] 1. Bench allocations: 57 allocs / 312 KB (not alloc-bound)
- [x] 2. CPU profile: **workAt scan** `RelationshipsByType::next` ~45% (scan each FoF's
  out-rels, dominated by `knows`, for the few `workAt`) + the sort ~22%
- [x] 3. Optimize: top-10 heap by (wf asc, plid asc, name desc) — plid/name once per row
- [x] 4. Re-bench: 50 allocs / 289 KB
- [x] 5. Wall-clock A/B: ~4.17 → ~3.50 ms (~16%), same window
- [x] 6. Value-identity: IC emit byte-identical

## Notes

The dominant cost (~45%) is `relationships(p, Outgoing, &["workAt"])` scanning
each ≤2-hop person's whole out-rel block (mostly their `knows`) to find the few
`workAt` rels — inherent to the per-node-mixed-type CSR, not cheaply avoidable.
The avoidable part was the sort: replaced collect-all-then-sort (a `pi64(plid)` +
`pstr(name)` per comparison) with a size-10 heap resolving plid/name once per
matching row. Value-identical (tied rows share the same projected (plid, name, wf)).

**Status: done** (sort heap; workAt scan is inherent).
