## Notes

The deep per-friend nesting (comments → replyOf → parent's hasTag → hasType class
check) is ~45% and inherent. The avoidable cost: `tags.insert(name.to_string())`
allocated a `String` per qualifying-tag *occurrence*, and the BTreeSet dedup'd
only *after* the allocation — so a tag seen N times wasted N−1 `to_string()`s.
Collect tag ids in a `HashSet<u32>` (dedup in the loop, no String) and resolve
names only for the 20 friends actually returned (sorted). 6.5× fewer allocations,
~17% faster, value-identical.

Follow-up: `tag_ids` was still a *fresh* `HashSet` per friend (moved into `rows`).
Reuse one set across friends (`clear()`) and snapshot only qualifying friends'
ids into a small `Vec<u32>`. **Bytes halved (303 KB → 144 KB)**, allocs 1,414 →
1,039; timing unchanged (alloc, not CPU). This measured the split: ~half the
~300 KB was the HashSets' hash-table overhead (working state), the rest is the
inherent result (top-20 tag-name strings + snapshot Vecs) — at the floor now.
