# 110 — Optimize IC10: friend recommendation

Per-query optimization loop. Methodology + commands:
[100_ic_perf_methodology.md](100_ic_perf_methodology.md).

## Loop
- [x] 1. Bench allocations: 52 allocs / 330 KB (not alloc-bound — `.any()` avoids per-post collect)
- [x] 2. CPU profile: bfs + CSR scoring walk + the sort (a `plid` lookup per comparison)
- [x] 3. Optimize: top-10 heap (plid once) + hoist bmon/bdom/plid columns
- [x] 4. Re-bench: 44 allocs / 288 KB
- [x] 5. Wall-clock A/B: ~3.36 → ~2.84 ms (~15%), same window
- [x] 6. Value-identity: IC emit byte-identical

## Measurements
| metric                  | baseline | after |
|-------------------------|----------|-------|
| allocs                  | 52       | 44    |
| bytes                   | 330,204  | 288,516 |
| wall-clock median (ms)  | ~3.36    | ~2.84 (~15%) |
| hot fn (CPU %)          | bfs + scoring + sort | bfs + scoring |

## Notes

Modest win — IC10 was already fast (wins ~370× vs Kùzu) and its cost is spread
across the 2-hop bfs, the per-FoF post-scoring walk, and the sort. Addressed the
two avoidable parts: a size-10 heap (resolve `plid` once, not per comparison) and
a hoist of the bmon/bdom (birthday filter) + plid columns (direct `Column::get`
instead of `pi64` re-resolving the key per FoF). The bfs + scoring traversal are
inherent. Value-identical.
